# Интерпретируемый кредитный скоринг

Система оценки кредитоспособности на базе логистической регрессии. В отличие от нейросетей, каждое решение модели математически объяснимо — это требование реального банковского регулирования.

**Результат:** ROC-AUC 0.836 / Gini 0.672 / скоркарта 21–775 баллов.

## Данные

Датасет **Give Me Some Credit** (Kaggle). Каждая строка — один заёмщик, 10 признаков.

| Признак | Описание |
|---|---|
| `SeriousDlqin2yrs` | Целевая переменная: 1 = дефолт, 0 = нет |
| `RevolvingUtilizationOfUnsecuredLines` | Доля использованного кредитного лимита (0–1) |
| `age` | Возраст заёмщика в годах |
| `NumberOfTime30-59DaysPastDueNotWorse` | Количество просрочек 30–59 дней |
| `DebtRatio` | Отношение долга к доходу |
| `MonthlyIncome` | Ежемесячный доход в долларах |
| `NumberOfOpenCreditLinesAndLoans` | Количество открытых кредитов |
| `NumberOfTimes90DaysLate` | Количество просрочек 90+ дней |
| `NumberRealEstateLoansOrLines` | Количество ипотек |
| `NumberOfTime60-89DaysPastDueNotWorse` | Количество просрочек 60–89 дней |
| `NumberOfDependents` | Количество иждивенцев |

**Дисбаланс классов:** 93.3% надёжных / 6.7% дефолтников.

## Пайплайн

### 1. Чистка данных
- Удаление строк с `age == 0` (ошибка ввода)
- Замена артефактного значения `96` на `NaN` в столбцах просрочек
- Винзоризация `RevolvingUtilization` до 1.0 (физический максимум)
- Обрезка `DebtRatio` и `MonthlyIncome` по 99-му перцентилю

### 2. Заполнение пропусков
- `MonthlyIncome` — медиана по возрастным группам (20% пропусков)
- `NumberOfDependents` — медиана (2.6% пропусков)

### 3. Feature Engineering
- Объединение трёх коллинеарных столбцов просрочек в один `TotalLatePayments` (корреляция между ними 0.975–0.987)
- Масштабирование через `StandardScaler` (fit только на train)

### 4. Модель
- `LogisticRegression(C=1, class_weight='balanced')`
- `class_weight='balanced'` компенсирует дисбаланс классов без SMOTE
- Подбор `C` через `GridSearchCV` (5-fold, scoring=roc_auc) — оптимум при C=1

### 5. Скоркарта
- Перевод вероятности дефолта в баллы по формуле: `Score = Offset + Factor × ln(Odds)`
- PDO=20, базовый балл=600, базовые шансы=1/19
- Итоговая шкала: 21–775 баллов, выше = надёжнее

## Результаты

| Метрика | Значение |
|---|---|
| ROC-AUC | 0.8362 |
| Gini Index | 0.6725 |
| Средний скор надёжных заёмщиков | 705 |
| Средний скор дефолтников | 653 |

### Интерпретация коэффициентов (Odds Ratios)

| Признак | Odds Ratio | Интерпретация |
|---|---|---|
| `TotalLatePayments` | 16.59 | Каждая просрочка увеличивает риск в 16.6 раз |
| `RevolvingUtilization` | 2.33 | 100% лимита кредитки → риск ×2.3 |
| `age` | 0.78 | Каждый год возраста снижает риск на 22% |
| `MonthlyIncome` | 0.85 | Выше доход → ниже риск |
| `NumberOfOpenCreditLinesAndLoans` | 1.18 | Слабый положительный эффект |

### Ориентиры по баллам

| Балл | Решение |
|---|---|
| > 700 | Низкий риск — лучшие условия |
| 600–700 | Приемлемый риск — стандартные условия |
| 400–600 | Средний риск — требует проверки |
| < 400 | Высокий риск — отказ |

## Ключевые решения

**Почему логистическая регрессия, а не XGBoost?** Банковское регулирование требует объяснимости каждого решения. Логрегрессия даёт прямую интерпретацию через коэффициенты и Odds Ratios.

**Почему `class_weight='balanced'`, а не SMOTE?** Для линейных моделей взвешивание классов даёт сопоставимый результат при меньшей сложности пайплайна и отсутствии риска утечки данных.

**Почему объединили столбцы просрочек?** Корреляция 0.975–0.987 между тремя столбцами создавала коллинеарность — один коэффициент уходил в отрицательную зону вопреки логике. После объединения в `TotalLatePayments` Gini вырос с 0.6685 до 0.6725.

**Почему C=1, а не C=100 (лучший по GridSearch)?** Разница в AUC составила 0.0001 — статистический шум. C=1 даёт более сильную регуляризацию и надёжнее обобщается на новых данных.
